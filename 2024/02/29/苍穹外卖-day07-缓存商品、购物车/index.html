苍穹外卖-day07-缓存商品、购物车

发表于 2024-02-29 | 分类于 苍穹外卖 |
字数统计: 4k | 阅读时长 ≈ 19

需求原因

通过小程序可以展示菜品和套餐数据，这些都是存储在同一个数据库当中，如果在某个时间段内，有大量的用户来点餐，就会频繁的查询数据库，数据库的访问压力相对来说比较大，造成数据库查询性能下降，用户使用体验不是很好，直观的体验是点击某个分类之后要经过几秒钟才能显示对应数据。

解决办法：使用缓存来解决，需要把商品数据缓存至redis当中。这会大大提高系统的查询性能。

购物车相关功能，点击+号，将商品添加到购物车。

缓存菜品

问题说明

从系统的查询性能分析问题，然后通过缓存的方式提高系统的查询性能。

  1. 是不是所有商品都要缓存？
  2. 商品缓存的粒度如何？

用户端小程序展示的菜品数据都是通过查询数据库获得，如果用户端访问量比较大，数据库访问压力随之增大。系统查询的瓶颈是数据库查询。

结果：系统响应慢、用户体验差

实现思路

通过Redis来缓存菜品数据，减少数据库查询操作。

Redis是基于内存访问数据的，数据库是磁盘IO操作。

内存操作相对于磁盘操作性能较高。

缓存逻辑分析：

  • 每个分类的菜品保存一份缓存数据（key：分类ID，value：分类下具体的菜品数据，使用string字符串保存，右侧的菜品在java中用list表示，我们可以将其序列化，转成redis的字符串）

  • 数据库中菜品数据有变更时清理缓存数据。（在商家端改了数据，要清理缓存，不然数据没有更新）

缓存套餐_Spring Cache

新的技术（spring-cache），由spring提供的技术。

Spring Cache

使用它可以进一步简化代码，提供了注解，要操作缓存数据，只要在方法上加入它提供的注解就可以。这种使用方式类似于事务管理。

Spring Cache是一个框架，实现了基于注解的缓存功能，只需要简单地加一个注解，就能实现缓存功能。

Spring Cache提供了一层抽象，底层可以切换不同的缓存实现，例如：

  • EHCache
  • Caffeine
  • Redis

导入maven坐标（不同缓存对应不同坐标，用redis就导入redis坐标）

常用注解：

注解 说明
@EnableCaching 开启缓存注解功能，通常加在启动类上
@Cacheable 在方法执行前先查询缓存中是否有数据，如果有数据，则直接返回缓存数据；如果没有缓存数据，调用方法并将方法返回值放到缓存中
@CachePut 将方法的返回值放到缓存中
@CacheEvict 将一条或多条数据从缓存中删除

实现思路

  1. 在启动类加上@EnableCaching注解，开启缓存注解功能。

  2. 使用@CachePut注解，将对应方法的返回值放入redis缓存中。

    该返回值的键，可以由#user.id或#result.id决定。

    UserController类

  3. 使用@Cacheable注解，如果查到数据，会把它直接放到缓存中。

    （Spring Cache底层是基于代理技术，即加入注解之后，Spring Cache会为当前的controller创建代理对象，我们在请求方法前是先进入代理对象，而在代理对象中查询redis）

    UserController类

  4. 使用@CacheEvict注解，方法结束完之后会删除缓存中的单条数据或全部数据

    UserController类

实现思路

具体实现思路如下：

  • 导入Spring Cache和Redis相关的maven坐标
  • 在启动类上加入@EnableCaching注解，开启缓存注解功能
  • 在用户端接口SetmealController的list方法上加入@Cacheable注解
  • 在管理端接口SetmealController的save、delete、update、startOrStop等方法上加入CacheEvit注解
